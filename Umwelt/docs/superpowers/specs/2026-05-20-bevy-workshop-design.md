# Bevy 3D 神经工坊 · 设计规格

> **状态**:草案 v0.1
> **日期**:2026-05-20
> **关系**:本文档定义"生物打印工坊"在 Umwelt 中的形态。上位参考 `docs/gameplay-architecture.md`(玩家循环 A 的核心场所),下位是后续的实现计划。

---

## 1. 动机

当前 HTML 的 2D 节点+连线编辑器已经到了表达力的极限,而工坊是整个游戏中玩家投入最多时间的核心界面 —— 它对应架构文档循环 A 的"修复"环节,也是循环 B 的"设计"环节。

把工坊迁到 Bevy 3D,顺便把它从"裸沙盒"升级为带 I/O 关卡的 Zach-like。HTML 在过渡期继续承担主世界沙盒(蚂蚁涌现观察场),工坊导出的模块通过 JSON 加载到 HTML 蚂蚁身上跑。长期目标是整个游戏迁到 Bevy,届时主世界与工坊统一。

本设计是迁移路径的第一步,只覆盖"工坊侧"。

---

## 2. 核心范式

**解剖空间**。Bevy 工坊里,3D 空间不是抽象布线区,也不是工作台台面 —— 而是**器官本身**。玩家在 3D 视图里看到的就是他正在设计的那个器官的物理形态。感受器细胞在它解剖学应该在的位置,中间神经元沿组织长度排布,输出在指定的脑端接口。空间位置 = 生物位置。

这个范式选择服从架构文档"诚实于神经结构的逻辑" —— 不做语义层的假抽象,具体硬件的具体物理位置就是它本身。

---

## 3. 自由度:全 DIY

工坊不强迫玩家做出"一个触须"。系统**只规定 I/O 契约**(关卡声明的输入受体类型、输出端口),玩家自由决定:

- **骨架形态** —— 任意 3D 包络。可以是触须、可以是分叉、可以是球状脑泡、可以是分布式神经丛。
- **感受器类型与位置** —— 从关卡的受体调色板里选,放在骨架的任意表面位置。
- **神经元数量** —— 在物理约束(见第 4 节)允许的范围内任意。
- **投射范围** —— 每个轴突的几何形态由玩家决定。
- **可塑性区域** —— 哪些连接可塑,绑定哪个 modulator,由玩家在 3D 里指定。

唯一被强制的是 I/O 端口的存在 —— 输出必须连到关卡声明的输出通道,受体必须是关卡允许的类型。中间的一切是玩家创作物。

**哲学**:玩家可能造出一个完全不像真实蚂蚁触须的东西,只要它能履行 I/O 契约。这是合法的也是被鼓励的 —— 真实生物的形态本身就是演化的解,不是设计的前提。

---

## 4. 物理约束(真实生物学)

工坊里的空间不是装饰。玩家做"把这个神经元放远一点"这一个决定,会同时撞上多组物理后果。这些约束基于真实昆虫神经组织的物理事实,而非游戏设计师虚构的"代价滑块"。

### 4.1 MVP 实化的物理

**几何**

- **体积预算**。器官包络是玩家在 3D 里画的任意形状,但有总体积上限。所有神经元胞体、轴突、突触都占体积。
- **感受器必须暴露**。化学感受器细胞必须落在器官包络的外表面(不能埋在内部)。机械感受器同理。这是关卡级别的硬约束。
- **双侧空间分辨率**。化学感受器必须有足够的空间基线才能解出方向。两个感受点之间距离越小,分辨率越差(在仿真层面这是 SNR / 浓度差的物理后果,不是数值惩罚)。

**能量**

- **代谢预算**。每个神经元的存在(维持静息电位)有持续代谢成本,每次 spike 有额外代谢成本,每个突触维持有代谢成本。器官有总代谢预算上限。
- **建造成本**。每个神经元在关卡内有一次性"建造成本"(对应真实生物的发育代价)。这个仅作总量限制,不在运行时影响行为。

**信号传导**

- **轴突有传导速度**。信号沿轴突传播有有限速度。距离 = 真实时延。
- **轴突粗细 ↔ 速度 ↔ 体积 ↔ 代谢**。这一组物理量耦合在一起。粗轴突 = 快传导 + 占体积 + 耗代谢。玩家通过调整轴突粗细做权衡。

### 4.2 数值的真实性策略

**有原则的延迟**(下文称"量级正确"策略):

1. 常数使用真实物理单位(μm³、pJ/spike、m/s),不用无单位"能量点"。
2. MVP 数值取在真实昆虫数据的一个量级之内,允许粗略估算。
3. 常数集中在 `constants/biology.rs`,每个常数附带文献来源注释(如"Attwell & Laughlin 2001,数量级估算")。
4. 后期精校时,改的是数字本身,不是单位制。

这避免了"先用整数后期换真实数据"造成的撕裂面积。

### 4.3 物理后果的玩家可见性

工坊 UI 必须把上述约束的当前消耗实时显示出来 —— 体积占用 / 总体积、代谢预算 / 总代谢、最长信号路径延迟。否则玩家无法做权衡。

## 5. 关卡形态:两段式测试

### 5.1 测试结构

**Stage 1 · 静态探针**

系统在玩家的器官周围摆一个静态化学场快照 —— 浓度场是 3D 的、固定的。玩家的受体按其在器官上的物理位置真的采样这个场。电路跑预定时长,系统记录输出波形并对照规范带。

特点:
- 决定性。同样的设计,同样的探针,同样的结果。可重放、可调试。
- 受体位置真的有意义。把双侧化学感受器堆在一起 vs 分开,采到的输入完全不同。
- 多场景。一关包含 N 个静态探针(梯度从左、从右、无梯度、双源、等),全部通过才算 Stage 1 完成。

**Stage 2 · 动态竞技场**

Stage 1 通过后解锁。系统把玩家的器官装到一只测试蚂蚁,放进一个小型 3D 测试环境(有化学源、有动态扩散场)。看蚂蚁在场景中的真实行为是否符合关卡描述。

特点:
- 验真实行为。"5 秒内到达 ChemA 源"这种行为级标准。
- 化学场动态演化。源的位置、扩散速率、衰减都是关卡参数。
- 多 trial。每个场景跑 N 次,要求成功率 ≥ 阈值。

**两阶段都通过 → 模块进入玩家的模块库(导出)。**

### 5.2 仿真器共用

Stage 1 的静态场和 Stage 2 的动态场使用**同一个化学场仿真器**。Stage 1 只是它的固定快照,Stage 2 是它的演化模式。实现上不是两套系统,是同一套系统的两种调用模式。

### 5.3 第一关:化学定向器官

**关卡 ID**:`chemotaxis-l1`
**关卡名**:化学定向器官 / Chemical Steering Organ
**输入受体调色板**:ChemA 化学感受器,ChemB 化学感受器(其他通道留给后续关卡)
**输出端口**:`turn_L`,`turn_R`,`forward`
**Stage 1 静态探针**(草案):
- probe-1:ChemA 梯度从左,无 ChemB → 期望 turn_L > 0.5 在 200ms 内
- probe-2:ChemA 梯度从右,无 ChemB → 期望 turn_R > 0.5 在 200ms 内
- probe-3:无任何梯度 → 期望 |turn_L − turn_R| < 0.1
- probe-4:ChemB 梯度(模拟危险物)从左 → 期望 turn_R > 0.5(避开)
- probe-5:ChemA 与 ChemB 同侧梯度 → 期望某种约定行为(可调,关卡设计点)

**Stage 2 动态场景**(草案):
- trial-1:单个 ChemA 源,任意方位 → 5 秒内蚂蚁距离源 < 阈值
- trial-2:单个 ChemB 源 → 5 秒内蚂蚁远离源
- trial-3:同时存在 ChemA 源和 ChemB 源 → 蚂蚁靠近 ChemA、远离 ChemB

具体阈值与时长在实现阶段调。这一节的目的是说明"关卡的具体构成是什么形状",不是定数。

## 6. MVP 作用域

**作用域:裸关卡 + 模块导出。**

具体包含:

- Bevy 工坊运行时:3D 场景、相机控制、节点放置、连线、轴突几何编辑、可塑性指定。
- 第一关 `chemotaxis-l1` 完整可玩:加载关卡 → 设计器官 → 跑 Stage 1 → 跑 Stage 2 → 通过 → 导出模块 JSON。
- 化学场仿真器(供 Stage 1 / Stage 2 使用)。
- 物理约束实时显示(体积、代谢、最长延迟)。
- 模块 JSON 格式定义(见第 7 节),与 HTML 侧可加载。

**不包含**(post-MVP):

- 关卡选择菜单 / 多关卡管理 / 模块库 UI。MVP 只需一关。
- 工坊壳的导航结构。启动直接进第一关。
- 元件库系统(架构文档循环 B)。MVP 默认所有节点类型可用。
- 主世界(主世界仍由 HTML 承担)。
- 化学场仿真器的 WASM 共用(HTML 继续用自己的)。

这个边界让 MVP 验证的是单一问题:**「3D 解剖工坊 + 真实物理 + 两段式测试」这套核心循环是否好玩**。其他都是后续工坊壳的事,等核心循环验证通过再做。

## 7. 工坊 ↔ HTML 主世界 的合约

### 7.1 导出格式

通过关卡后,工坊导出一个 JSON 模块。结构(草案):

```jsonc
{
  "schema": "umwelt-module-v1",
  "level_id": "chemotaxis-l1",
  "compiled_at": "<iso8601>",

  // 受体在蚂蚁身体坐标系下的位置(给 HTML 主世界的化学场采样用)
  "receptors": [
    { "id": "r0", "type": "chem_A", "body_pos": [x, y, z], "cone_angle": ..., "cone_range": ... },
    ...
  ],

  // 编译后的电路图。同 HTML 现有 NeuralGraph 序列化格式,扩展一个 edge.delay_ms 字段
  "graph": { ... },

  // 元数据(给 HTML UI 显示,不影响运行)
  "meta": {
    "volume_used_um3": ...,
    "metabolic_cost_pj_s": ...,
    "max_path_delay_ms": ...
  }
}
```

### 7.2 编译策略:A-lite

工坊里的真实物理在导出时**部分编译进 HTML 运行时**:

- **传导延迟**:每条轴突的物理长度按真实传导速度算成 `delay_ms`,作为 `edge.delay_ms` 字段进入导出图。**这是 HTML 运行时唯一需要新支持的概念**。HTML evaluator 给每条边加一个环形缓冲实现延迟。
- **体积、代谢预算**:仅在工坊里作为"关卡通过的门槛"履行。一旦通过、装载到 HTML 蚂蚁身上,这些预算不再影响运行时。它们以元数据形式跟着模块走,供 HTML UI 显示,但不参与仿真。

### 7.3 为什么是 A-lite

完整答案在终态:游戏长期愿景是全 Bevy 化,届时 3D 物理在运行时完整存在(对应 brainstorm 里的方案 B)。HTML 是过渡期的临时主世界,任何为 HTML 添加的运行时逻辑最终都会随 HTML 一起被替换。

但**传导延迟**是玩家在工坊里调的设计决定**直接表现在主世界行为**的关键单一线索 —— "我把两个神经元放近了,蚂蚁反应更快"。如果 HTML 不实现 delay,玩家会发现"近放远放装上之后没区别",工坊里的空间设计在主世界就完全失声。

体积和代谢不一样。它们的作用在工坊设计阶段已经发生(塑造了玩家的设计选择),装载后玩家不会再编辑导出的电路,所以预算在运行时只是死数据。HTML 阶段不为它们写运行时逻辑是合理的取舍 —— 这笔投资最终会随 HTML 被丢。

**简言之**:为 HTML 保留"距离 → 延迟"这一根诚实链,其他真实物理在 HTML 阶段牺牲。等迁到 Bevy 时全部活过来。

### 7.4 HTML 侧的局部改动

- `NeuralGraph` 边结构加 `delay_ms` 字段(默认 0,向后兼容老存档)。
- evaluator 实现环形缓冲。
- 存档 schema v9 → v10(增加 edge.delay_ms,以及可选的 module-loaded 元数据块)。
- 主世界 UI 加一个"装载模块"入口,接受工坊 JSON 导出。

主世界其他部分(化学场、蚂蚁运动学、可塑性更新规则)**不动**。

## 8. 开放问题

按粗略优先级:

**3D 编辑器的交互模型**。鼠标 + 键盘下,如何放置 3D 神经元、如何在 3D 里拉轴突、如何选择重叠的对象、相机控制策略。这是单独需要一个 UX 设计回合的问题,在实现计划阶段细化。

**轴突几何编辑的粒度**。轴突是直线段?可弯折的折线?贝塞尔曲线?粗细是单一参数还是沿长度变化?直接影响"传导速度"如何计算。

**"器官包络"如何在 3D 里被画出来**。玩家是先画一个壳再往里放神经元,还是放完神经元自动生成最小包络,还是关卡预设包络让玩家在其中布局?

**Stage 1 静态探针的化学场如何构造**。是关卡作者手工指定的浓度分布,还是参数化生成(梯度方向、强度、衰减),还是从 Stage 2 动态场景里截取的快照?

**通过失败的诊断反馈**。玩家电路过不去某个 probe,系统应该告诉他什么?(原始波形对比?差异分析?提示哪个 probe 失败到什么程度?)

**模块版本演化**。玩家把同一关重做(优化器官),旧导出的模块如何处理?替换?并存?

**第二关、第三关的形态预想**。MVP 只做一关,但下一关大概会是什么?(机械感受器避障?可塑性学习关?)预想会反过来约束 MVP 的关卡格式设计。

**Bevy 项目和现有 HTML 项目的代码组织**。两者是同一个 git repo 的子目录,还是分仓?Rust workspace 的层次?

**真实生物学常数的源**。MVP 数值"量级正确"具体取哪些文献作来源?(Attwell & Laughlin 2001 是脊椎动物估算,昆虫尺度的源需要另查。)

**许可证与第三方依赖审查**。Bevy 是 MIT/Apache 双许可,游戏本体的许可证策略尚未确定。

## 9. 设计宪法检查

按架构文档第 14 节附录:

1. **是否引入了系统级的功能语义?** 否。关卡的 I/O 契约用的都是物理通道(ChemA 受体、turn_L 端口),没有"避障检测器"这种功能命名。
2. **是否基于虚构的生物学机制?** 否。物理约束(体积、能量、传导速度、双侧 SNR)都是真实昆虫神经组织的物理事实。
3. **是否把有 Umwelt 的主体和无 Umwelt 的工具混同?** 否。工坊本身是基础设施(工具),其中设计的器官是主体的一部分。分界清晰。
4. **是否用数值缩放替代结构性差异?** 否。后续关卡的差异来自结构性新元素(新受体类型、新输出端口、新可塑性要求),不是缩放参数。
5. **是否给硬核或新手之一增加摩擦而另一方无收益?** 全 DIY 范式对新手摩擦更大(没有骨架模板),需要在 UI 上提供"半透明参考"(架构文档 5.1 的默认蚂蚁观察模式)来降低门槛。这一点在 MVP 之后细化。
6. **是否让玩家设计能力之外的东西成为核心进度变量?** 否。通过关卡的依据是设计能力(电路是否能履行 I/O 契约),不是资源积累或时间投入。
